<DropCap>H</DropCap>andoff — это не момент в пайплайне. Это контракт между дизайнером и разработчиком, который пишется задолго до финального макета. Good handoff — это когда разработчик не задаёт вопросов не потому что боится, а потому что всё ясно: компоненты названы так же, как в коде; цвета лежат в переменных, у которых те же имена, что и в CSS; отступы выстроены по шкале, а не на глаз; edge cases подписаны; responsiveness проговорена. Всё остальное — набор инструментов, которые этот контракт обслуживают.

Если смотреть на handoff как на момент, его почти невозможно сделать хорошо: за последние три дня спринта дизайнер не успеет ни привести в порядок структуру, ни договориться о названиях, ни написать спецификацию. Если смотреть как на протяжённый процесс — оказывается, что инструменты готовы помочь, а провалы происходят по понятным причинам: никто не договорился, как называть кнопки, никто не ведёт токены, никто не сидит рядом во время имплементации. Эта тема — карта этих причин и инструментов.

<SectionBreak id="01" label="FIGMA DEV MODE" />

Dev Mode Figma анонсировала на Config 2023 в июне 2023 года — мероприятии, которое за последние годы стало главной точкой релизов платформы. Dev Mode пришёл на смену старой «Inspect»-панели, которую разработчики использовали в режиме view-only: раньше они открывали макет, кликали на элемент и читали его свойства в правой колонке. Dev Mode формализовал этот сценарий в отдельный интерфейс, специально спроектированный под задачи имплементации.

Что в нём есть. Во-первых, **read-only режим**: разработчик видит макет, может навигировать, но не редактирует. Это не ограничение, а защита — ни один разработчик не хочет случайно двигать иерархию дизайнера, пока разбирается в отступах. Во-вторых, **Inspect panel** — полный набор измерений: dimensions, spacing, typography, colors, effects, constraints. В-третьих, **code snippets** — автоматически генерируемые фрагменты кода: CSS для веба, Swift для iOS, XML для Android. Это boilerplate для быстрого старта, а не production-ready код; о причинах — чуть ниже. В-четвёртых, **diff между версиями**: Dev Mode показывает, что изменилось в файле после предыдущего «snapshot», и разработчик видит точечные правки, а не перечитывает весь макет заново. В-пятых, **Ready-for-dev status** — дизайнер помечает frame как готовый к разработке; пока флаг не поднят, разработчику нечего смотреть.

<MarginNote>
Dev Mode — Figma Config 2023 (июнь). Начиная с 2024 года Dev Mode тарифицируется как отдельная платная надстройка — у разработчиков нет полного дизайнерского доступа, а место обходится заметно дешевле полноценного editor seat. Конкретные цифры со временем меняются; актуальные смотреть в Figma Pricing.
</MarginNote>

Практический смысл: Dev Mode переводит разработчика из роли «случайного гостя в чужом инструменте» в роль первого класса. До 2023 разработчик смотрел на Figma как на чужую территорию — сегодня у него свой режим, свои сниппеты, свой статус у фрейма. Это меняет и ритм коммуникации: дизайнер помечает готовые frame флагом, разработчик подписан на канал изменений, никто не ждёт письма «окей, можно брать в работу».

<SectionBreak id="02" label="ЧТО DEV MODE НЕ ДЕЛАЕТ" />

О Dev Mode часто говорят так, будто он закрывает задачу handoff целиком. Это не так. Его способности заканчиваются ровно там, где начинается настоящая разработка.

Dev Mode **не генерирует production code**. Сниппеты, которые он выдаёт, — это CSS, построенный по свойствам конкретного frame: `padding: 12px 16px; background: #D8412F; border-radius: 8px`. В production-проекте эти значения должны быть не хардкодом, а ссылками на токены: `padding: var(--space-2) var(--space-3); background: var(--color-brand-primary); border-radius: var(--radius-sm)`. Copy-paste из Dev Mode в реальный код — это анти-паттерн: через месяц вы получите полотно захардкоженных пикселей, которое не реагирует на смену темы и rebrand.

Dev Mode **не разрешает responsive автоматически**. Auto Layout во Figma — это не Flexbox-код. Разработчик сам решает, как перенести `hug contents` и `fill container` в `flex`, `grid`, `min-width`, `max-width` и медиа-запросы. Сниппет покажет, что элемент занимает 100% ширины — но никогда не расскажет, что на мобильном он должен переходить в колонку.

Dev Mode **не синхронизирует токены с кодом**. Он показывает имя переменной из Figma Variables (`colors/brand/cinnabar-500`), но ответственность за то, чтобы это имя совпадало с `--color-brand-cinnabar-500` в CSS, лежит на пайплайне токенов — а его настраивают отдельно.

Dev Mode **не заменяет разговор**. Никакая панель свойств не расскажет про edge cases: что делает компонент при пустом состоянии, как ведёт себя при ошибке сети, как выглядит на маленьком экране в правом ретурн-flow. Эти вещи проговариваются голосом или в тексте, не в инструменте.

<SectionBreak id="03" label="DESIGN TOKENS" />

Design token — это именованная абстракция дизайн-решения. Цвет `#D8412F` превращается в токен `color.brand.cinnabar.500`; отступ 16px — в `space.md`; радиус 8px — в `radius.sm`. Суть токена — не в том, что он хранит значение, а в том, что он даёт этому значению имя, к которому можно обращаться из двух инструментов сразу — Figma и кода.

Зрелый дизайн-токен-слой состоит из трёх уровней. **Primitive tokens** — сырые значения: вся палитра (`color.red.100` до `color.red.900`), вся шкала отступов (`space.0` до `space.16`), все радиусы, все durations. Primitive — это словарь, где каждое имя один к одному соответствует значению. **Semantic tokens** — роли: `color.background.primary`, `color.text.muted`, `color.border.subtle`. Semantic-токен не хранит значение сам, а ссылается на primitive: `color.background.primary = {color.brand.cinnabar.500}`. **Component tokens** — токены уровня компонента: `button.bg.primary`, `button.text.primary`, `input.border.focus`. Они тоже ссылаются — обычно на semantic.

Три уровня выглядят как избыточность, пока не случается первый rebrand. Меняется cinnabar-500 с `#D8412F` на `#C7401E` — и весь интерфейс обновляется сам: semantic остались прежними, component остались прежними, поменялся один primitive, и каскад прокатился дальше. Без semantic-слоя пришлось бы руками перепривязывать все компоненты, которые использовали этот цвет как фон, как акцент, как hover. Semantic — это буфер между визуальным решением и ролевой моделью.

<MarginNote>
DTCG (Design Tokens Community Group) — W3C Community Group, работающая над форматом токенов с 2019–2020 годов. Community Group публикует community reports, а не W3C Recommendations — это разные треки стандартизации. На практике формат де-факто принят Figma Variables, Style Dictionary, Token Studio и большинством инструментов вокруг.
</MarginNote>

<SectionBreak id="04" label="DTCG СПЕЦИФИКАЦИЯ" />

Чтобы токены были переносимы между инструментами, им нужен общий формат. Этим занимается **Design Tokens Community Group** — W3C Community Group, в которой участвуют Adobe, Figma, Microsoft, Google, Salesforce и десятки других. Группа выпускает community reports; с 2019–2020 годов в работе находится draft-спецификация JSON-формата для токенов.

Что задаёт draft. **Тип** токена: `color`, `dimension`, `duration`, `fontFamily`, `fontWeight`, `number`, `cubicBezier` и так далее. Тип — это не просто метка, а контракт о том, как инструмент должен интерпретировать значение. **Value** — сама величина: строка, число, объект. **Reference syntax** — ссылки на другие токены через фигурные скобки: `{color.brand.primary}`. **Aliases** — специальный случай ссылки, когда один токен целиком указывает на другой. **Composite tokens** — сложные объекты: `typography` как bundle из `fontFamily`, `fontSize`, `fontWeight`, `lineHeight`, `letterSpacing`.

Community Group не идёт по треку W3C Recommendation — её результатами являются community reports, и в текущем драфте остаются спорные места, например точное описание modes и темизации. Но фактически именно на этом формате сегодня сходятся Figma Variables (экспорт в DTCG JSON), Style Dictionary (поддержка DTCG-входа), Token Studio (основной формат). Если проект строится на токенах, формат имеет смысл выбрать сразу DTCG — риск, что завтра придётся мигрировать, минимален.

<SectionBreak id="05" label="TOKENS PIPELINE: FIGMA → CODE" />

Схема, которая за последние два года стала де-факто стандартом для команд среднего размера.

Первый шаг — **токены как Variables в Figma**. Дизайнер ведёт два слоя: primitive (палитра, шкалы) и semantic (роли). Component-токены в Figma часто опускают и выражают их через именование компонентов.

Второй — **Token Studio plugin**. Опенсорс-плагин, поддерживается Яном Сиксом (Jan Six) с 2020 года (раньше назывался Figma Tokens). Плагин читает Variables и экспортирует их в JSON по DTCG-формату. Альтернативно — прямой экспорт через Figma API в custom-скрипте, если вы хотите свою логику трансформаций.

Дальше всё уходит в код. **Style Dictionary** — опенсорс-инструмент от Amazon (2017, первый мейнтейнер Danny Banks) — берёт JSON с токенами и трансформирует его в любые платформенные форматы: CSS custom properties, SCSS, iOS Swift, Android XML, JS-константы, JSON для React Native; одна входная точка — десятки выходов. Репозиторий токенов обычно живёт как отдельный пакет (`@company/design-tokens`), на каждом merge CI прогоняет Style Dictionary, собирает билды под все платформы и публикует новую версию npm-пакета, которую frontend подтягивает следующим релизом. В итоге в коде появляются CSS-переменные и JS-константы с теми же именами, что у дизайнера в Figma: компонент пишет `background: var(--color-background-primary)`, дизайнер не рисует фон вручную, а привязывает к `color.background.primary`, и когда значение меняется в Figma, изменение проходит по всей цепочке за часы, а не за дни переписки.

Альтернативы существуют. **Specify** — коммерческая платформа (активна с ~2019), закрывает ту же задачу в SaaS-модели: подключаете Figma, получаете API и CLI, который раздаёт токены в код. **Supernova** — коммерческая платформа для дизайн-систем (2019), делает то же плюс документацию компонентов и сборку сайта с примерами. Выбор между DIY (Token Studio + Style Dictionary) и SaaS зависит от команды: первый путь дешевле и гибче, второй — быстрее стартует и меньше требует поддержки.

Главное — **единый источник истины**. Цвета в Figma не должны быть одним списком, а в коде другим: либо Figma диктует код через pipeline, либо наоборот — но два независимых списка всегда разъезжаются.

<SectionBreak id="06" label="CODE CONNECT" />

Code Connect — фича, которую Figma анонсировала на Config 2024 в июне 2024 года. Идея простая: связать Figma-компонент с production-компонентом в коде, чтобы в Dev Mode разработчик видел не абстрактный CSS, а готовый JSX (или Swift, или Kotlin).

Как это выглядит на практике. Дизайнер выбирает в макете instance `Button/Primary/Large`. В Dev Mode, помимо обычных свойств, появляется секция «Use in code» с конкретным сниппетом: `<Button variant="primary" size="large">Label</Button>`. Сниппет берётся не из автогенерации, а из mapping, который настроил команда: код-компонент декорируется метаданными `figma.connect(Button, 'figma-file-url/node-id', { props: {...} })`, и Figma теперь знает, какой React-компонент соответствует какому дизайн-компоненту.

Бенефит — прямой: разработчик копирует правильное использование, а не догадывается по CSS, какой именно `Button` из библиотеки имеется в виду. В большой дизайн-системе с десятком вариантов кнопок это снимает целый класс ошибок — не «синий прямоугольник», а именно `<Button variant="primary" />` с документированным API.

Ограничения — такие же прямые. Code Connect **требует ручной настройки** per component: mapping пишется для каждого компонента, который хотите связать. Арифметика простая: на зрелой библиотеке в 20 стабильных компонентов настройка занимает пару дней и окупается за недели — каждый handoff становится короче на те самые догадки по CSS. На дизайн-системе в 200 компонентов разовая настройка — уже работа на недели, и если компоненты ещё перекраиваются каждую неделю, mapping устаревает быстрее, чем приносит пользу, и начинает врать. На MVP это прямой overengineering; подключать Code Connect имеет смысл, когда имена компонентов в Figma и API в коде перестали расходиться чаще раза в квартал.

<SectionBreak id="07" label="NAMING CONVENTIONS" />

Если попробовать назвать одну причину, по которой handoff ломается, — это **несогласованные имена**. Инструменты тут бессильны: ни Dev Mode, ни Code Connect, ни pipeline не спасут, если дизайнер называет кнопку «Button Blue», а разработчик — `<Button variant="primary" />`.

Три уровня naming, которые нужно согласовать.

**Component naming**. Figma использует slash-конвенцию для variants: `Button/Primary/Large` разбирается на property `variant=primary, size=large`. Правильный паттерн — прямое соответствие: имя во Figma и API компонента в коде читаются одинаково. `Button/Primary/Large` ↔ `<Button variant="primary" size="large" />`. Это не просто косметика: когда имена совпадают, Code Connect настраивается за минуты, критика и review проходят без перевода, а джун в команде быстрее собирает mental model.

**Token naming**. Имя токена должно читаться одинаково в Figma и в CSS. Figma variable `spacing/md` ↔ CSS `--space-md` (или `--spacing-md` — главное, заранее договорились). Разрыв возникает, когда в Figma токен `spacing/md = 16`, а в коде — `padding: 16px` без переменной. Через три спринта вы не помните, какой из десяти 16px у вас токен, а какой — случайность.

**Layer naming**. Слои в Figma — тоже имена, хотя про них часто забывают. Frame `Frame 1247` — анти-паттерн; frame `Header` — семантика, которая у разработчика превратится в `<Header />`. Это не значит, что каждый элемент нужно называть вручную, — дизайнер отбирает имена для тех слоёв, которые имеют смысл за пределами макета: секции, контейнеры, компоненты-композиции.

<MarginNote>
Самый частый анти-паттерн: дизайнер называет компоненты случайно («Button Blue»), разработчик переименовывает в коде под свои конвенции («PrimaryButton»), потом дизайнер дорабатывает, опять называя по-своему, — имена drift apart на каждой итерации. Через полгода Figma-файл и репозиторий — как два разных языка с общими идиомами.
</MarginNote>

Правило простое: **naming-конвенция — это часть дизайн-системы, а не приватный стиль дизайнера или разработчика**. Она документируется, обсуждается, применяется ко всему — от токенов до layer names.

<SectionBreak id="08" label="HANDOFF RITUAL" />

Инструменты — половина. Вторая — ритуалы, без которых инструменты остаются полочным украшением.

**Design critique перед handoff**. Прежде чем отдавать макет в разработку, он проходит внутренний ревью дизайн-команды — и это момент, когда ловятся проблемы, которых разработчик никогда не увидит: несогласованность с системой, пропущенные состояния, неправильные токены. Дешевле исправить в Figma, чем в production.

**Walk-through с разработчиком**. Живая сессия на 15–30 минут, где дизайнер ведёт разработчика через макет: «вот основной flow, вот edge cases, вот где я сомневался, вот где у нас до сих пор открыт вопрос». Это в разы быстрее, чем три дня комментариев в Figma по одному вопросу в час.

<MarginNote>
Walk-through — не формальная встреча «на весь спринт». Это короткая сессия на конкретный feature, желательно в тот же день, когда дизайнер помечает frame как ready-for-dev. Через сутки контекст у дизайнера уже размылся.
</MarginNote>

**Spec-документ для сложных flow**. Не каждый макет нужен со спецификацией, но для многошаговых flow — оформление заказа, воронка регистрации, инлайн-валидация формы — текстовое описание незаменимо. Edge cases, error states, условия переходов, responsive behavior. Страница в Notion или Linear с таблицей состояний экономит недели переделок.

**QA вместе**. Handoff не заканчивается в момент, когда дизайнер нажал «готово». Дизайнер остаётся в Pull Request review: проверяет, что имплементация совпадает с макетом в мелочах — отступы, hover-состояния, поведение при overflow. «Готово, закрыть macbook» — анти-паттерн: без дизайнерского QA в продакшн уходит «то же самое, но хуже».

**Feedback loop**. После релиза дизайнер и разработчик обсуждают, что было сложно: где макет был неполным, где компонент вёл себя неожиданно, где документация отставала. Этот разговор — топливо для следующего цикла: конвенции донастраиваются, spec шаблон дописывается, компонент фиксится.

<SectionBreak id="10" label="СИНТЕЗ" />

Handoff — это не «передать Figma-файл и пойти домой». Это партнёрство дизайнера с разработчиком, в котором дизайнер готовит среду для быстрой и правильной реализации, а разработчик получает не набор скриншотов, а работающий контракт. Tokens дают общий словарь значений; Dev Mode — общий интерфейс для имплементации; Code Connect — общий язык компонентов; naming conventions — общую грамматику; ритуалы (critique, walk-through, spec, QA, feedback loop) — общее время и внимание.

Инструменты без ритуалов не работают: Dev Mode без walk-through превращается в очередной view-режим, где разработчик гадает; токены без конвенций — в два независимых списка, которые разъезжаются. Ритуалы без инструментов тоже не работают: walk-through без Dev Mode — это рассказ по скриншотам, а spec без токенов — хардкод пикселей, который ломается при первом rebrand. Четыре инструмента и пять ритуалов — это один процесс, и сила handoff в том, насколько полно команда его воспроизводит каждую неделю, а не на красивой презентации в конце квартала.
