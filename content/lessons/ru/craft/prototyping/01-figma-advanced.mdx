<DropCap>Р</DropCap>азница между дизайнером, который «знает Figma», и дизайнером, который ей владеет, — это разница между ста компонентами в библиотеке и теми же ста компонентами, но с правильно устроенными переменными, чистым Auto Layout и interactive states. Инструмент один, результат — разный на порядок. Один макет в конце квартала превращается в тяжёлый файл с ручной подменой цветов; другой живёт как дизайн-система, где смена brand-палитры занимает тридцать секунд, а прототип выглядит как настоящее приложение с состояниями и логикой.

Продвинутый Figma — это не набор трюков, а смена модели мышления. Вы перестаёте думать фреймами и начинаете думать компонентами, токенами и контрактами. Каждый элемент становится ответом на два вопроса: как он адаптируется к контексту (Auto Layout) и из каких переменных он собран (Variables). Остальное — Component Properties, Interactive Components, Dev Mode — надстраивается над этим фундаментом. Ниже — карта этих слоёв.

<SectionBreak id="01" label="AUTO LAYOUT" />

Auto Layout появился в Figma в конце 2019 года и прошёл несколько итераций, но настоящий скачок случился в июне 2022, на Config 2022 — когда добавили wrapping, min/max constraints и absolute positioning. С этого момента Auto Layout перестал быть «удобством для кнопок» и стал Figma-аналогом CSS Flexbox: мощной адаптивной моделью, на которой должен быть построен каждый компонент.

<MarginNote>
Figma Config 2022 (июнь 2022) — релиз Auto Layout v3 с wrapping и min/max. Это момент, после которого абсолютное позиционирование внутри компонентов превратилось в anti-pattern.
</MarginNote>

Базовая механика: вы выбираете группу элементов и превращаете её в Auto Layout фрейм. Фрейм получает три свойства — direction (horizontal / vertical / wrap), spacing между детьми (gap) и padding по четырём сторонам. Это и есть Flexbox в Figma: `flex-direction`, `gap`, `padding`. Wrap (перенос строк) появился на Config 2022 и приблизил Auto Layout к `flex-wrap: wrap`.

Важнее эргономики — **resizing modes**. Каждый элемент внутри Auto Layout фрейма имеет три режима ресайза: **fixed** (фиксированный размер), **hug contents** (обёртывает содержимое) и **fill container** (занимает доступное пространство). Это и есть правильная ментальная модель компонента: кнопка обычно — `hug` по ширине и `hug` по высоте; заголовок в карточке — `fill` по ширине и `hug` по высоте; сама карточка — `fill` по ширине родителя и `hug` по сумме детей. Стоит один раз выстроить эту логику на всём файле, и компоненты начинают вести себя предсказуемо в любых контекстах: в сайдбаре, в модалке, на мобиле.

Поверх resizing modes лежат **min/max constraints**. Кнопка может быть `hug` по тексту, но иметь `min-width: 120px`, чтобы короткие лейблы («OK», «Да») не превращались в квадратики. Карточка может быть `fill` по ширине, но иметь `max-width: 480px`, чтобы на широком экране не растягиваться до нечитаемой строки. Min/max — это то, что превращает Auto Layout из «раскладывает по оси» в «строит адаптивный компонент».

Отсюда первое жёсткое правило: **каждый компонент в библиотеке должен быть Auto Layout**. Абсолютное позиционирование внутри компонентов — anti-pattern почти во всех случаях; исключения — декоративные оверлеи, badge-точки, некоторые иконки. Если вы видите компонент, собранный абсолютным позиционированием, знайте: через месяц в нём сломается выравнивание, через два — кто-то передвинет элемент на полпикселя, и к концу квартала компонент превратится в минное поле. Auto Layout закрывает эту утечку энтропии.

<SectionBreak id="02" label="VARIABLES" />

Variables — главное событие Figma за последние годы. Анонс на Config 2023 (июнь 2023), релиз в середине того же года. До Variables дизайн-токены в Figma жили в стилях (Color Styles, Text Styles) и плагинах; после Variables они впервые получили нативную модель — с типами, scope и modes.

Variables бывают четырёх базовых типов. **Color** — цветовые токены: `colors/brand/cinnabar-500`, `colors/surface/elevated`. **Number** — числа: spacing, radius, font-size, line-height в абсолютных единицах. **String** — строки: лейблы кнопок, тексты, а также язык для прототипа-локализации. **Boolean** — `true/false`, используется для conditional visibility: `is_loading`, `is_error`, `is_premium_user`. В 2024 к ним добавились **typography variables** — bundle из font-family, weight, size и line-height, закрывающий последний зазор между Figma и design-tokens спецификациями.

<MarginNote>
Typography variables — Config 2024 (июнь 2024). До этого шрифтовые токены жили в Text Styles, которые не умели в modes и aliases. После 2024 font stack стал таким же first-class токеном, как цвет.
</MarginNote>

Второй фундаментальный механизм — **scopes**. Каждая переменная может ограничивать, где её разрешено использовать: только как fill, только как stroke, только как text color, только как radius. Это звучит избыточно, пока вы не попробуете большой файл без scope: дизайнер случайно применяет `colors/border/subtle` как фон кнопки, и через месяц вы разбираете, почему половина interface получила серый фон. Scopes — это типизация для токенов.

Третий — **modes**. Одна и та же переменная `colors/surface/primary` может иметь разные значения в разных модах: `#FFFFFF` в light и `#0B0B0F` в dark. Когда вы переключаете моду на фрейме, все переменные автоматически принимают соответствующие значения. Моды закрывают классические кейсы: light/dark, brand A / brand B, density compact/comfortable, language EN/RU с разными типографическими метриками. До Variables всё это решалось через дубликат фреймов или сложные плагины; теперь — одной сменой моды.

Четвёртый — **aliases**. Переменная может ссылаться на другую переменную: `colors/action/primary` ссылается на `colors/brand/cinnabar-500`. Это позволяет строить два слоя токенов: **primitive** (сырые значения: палитра, spacing scale) и **semantic** (что они означают в интерфейсе: primary action, danger, muted text). Semantic поверх primitive — это и есть design tokens по версии DTCG; Variables дают нативный механизм, раньше требовавший внешних плагинов.

Экспорт наружу — через плагины. Token Studio (бывший Figma Tokens, open source, поддерживается Яном Зиксом с 2020 года) умеет выгружать Variables в JSON, CSS, Swift, Android. Всё это кладётся на **DTCG format** — спецификацию Design Tokens Community Group, W3C draft, над которым работают Adobe, Figma, Microsoft и другие. Практический паттерн: Figma — единственный источник истины для токенов, CI подтягивает JSON в кодовую базу, разработчики работают с понятными CSS-переменными или Swift-constants.

<SectionBreak id="03" label="COMPONENT PROPERTIES" />

Component Properties и variants — два разных механизма, которые новички часто смешивают. **Variants** — это когда компонент имеет несколько вариантов всей формы целиком (Button/Primary/Large, Button/Secondary/Large, Button/Primary/Small) и вы переключаетесь между ними через property-свойство. **Component Properties** — это когда на уровне instance вы точечно меняете отдельные элементы внутри компонента, не трогая его вариант.

Четыре типа Component Properties:

**Boolean properties** — показать/скрыть элемент. Классика: иконка слева от текста кнопки. Вместо того чтобы делать варианты «с иконкой» и «без иконки», вы вешаете boolean `show_icon` на нужный элемент. На instance его можно переключить одним кликом.

**Instance swap properties** — замена nested instance. Тот же кейс иконки, но теперь — какая именно иконка. Вместо ста вариантов с разными иконками вы делаете один компонент с property `icon`, которая принимает любой instance из библиотеки иконок.

**Text properties** — override текста на instance level без detach. Раньше, чтобы изменить лейбл кнопки, нужно было либо полагаться на прямое редактирование текста (теряя возможность типизации), либо делать варианты под каждую подпись. Text property формализует эту замену: property `label` объявляется на компоненте и переопределяется на instance.

**Variant properties** — multi-select по осям. Size (S/M/L), state (default/hover/active/disabled/loading), emphasis (primary/secondary/ghost) — это оси, которые действительно разделяют «типы» компонента. Каждая ось = variant property; комбинации — это варианты.

Правильная граница между variants и non-variant properties критична для maintainability. Общее правило: **variants — это вещи, которые меняют визуальную структуру компонента** (размер, состояние, эмфаза); **non-variant properties — это вещи, которые варьируются на каждом instance** (текст, иконка, видимость отдельного элемента). Если положить текст кнопки в variants, у вас получится бесконечная комбинаторика из 500+ вариантов; если положить состояние в text property, пропадёт возможность переключать состояния в прототипе.

<SectionBreak id="04" label="INTERACTIVE COMPONENTS" />

Interactive Components — одна из самых недооценённых возможностей Figma. Анонсированы в июне 2022 на Config, публично released в сентябре 2022. Идея простая: вы описываете переходы между variants прямо внутри компонента, а не на уровне каждого экрана прототипа.

Раньше, чтобы прототипировать кнопку с hover-состоянием, нужно было вручную соединять стрелками два фрейма для каждого места, где эта кнопка появляется. В большом файле это означало тысячи связей. Interactive Components переносят поведение внутрь компонента: Button/Default → On Hover → Button/Hover, Button/Default → On Click → Button/Active, Button/Active → After Delay 800ms → Button/Loading. Один раз описали — везде работает.

Конкретный пример: кнопка с состояниями default / hover / pressed / loading / success. Внутри компонента — пять variants и четыре перехода: на hover переключается на hover, на click — на pressed и через 100ms на loading, через 2 секунды — на success, через ещё секунду — обратно на default. В прототипе это работает без единой внешней стрелки: вы просто вставляете кнопку и получаете живое поведение.

<MarginNote>
Interactive Components — Figma Config 2022 (июнь), публичный релиз — сентябрь 2022. До этого state-machine внутри компонента собирался только внешними стрелками прототипа.
</MarginNote>

Следующий слой — **conditional logic** (добавлена в 2023): блоки `if` в прототипе читают Variables и решают, куда идти. Это превратило Figma-прототип из fixed-flow в настоящий interactive mock. Минимальный рабочий пример — переключатель темы: на кнопке вешаем action `Set variable` → `is_dark = !is_dark`, а корневой фрейм экрана привязан к моде Variables, зависящей от `is_dark`. Один клик — и весь прототип, во всех вложенных компонентах, переключается на dark. Никакого detach, никаких дубликатов экранов.

Второй нетривиальный приём — счётчики и условные состояния корзины. Переменная `cart_count: number`, на кнопке «Add to cart» action увеличивает её на единицу, а badge иконки-корзины через conditional `if cart_count > 0 show` автоматически появляется и показывает значение. Это уже state-machine, не слайдшоу.

Ограничения честно назвать: это всё ещё прототип. Нет персистентного state между сессиями (перезапустили — всё сбросилось), нет сетевых запросов, нет настоящих данных. Figma прототипы — это отличный инструмент для user testing и презентации stakeholders, но не замена frontend-разработке. Граница между «prototype-grade fidelity» и «production» остаётся важной; overreach в сторону сложной логики обычно означает, что вы начали писать приложение в Figma — и это почти всегда плохая идея.

<SectionBreak id="05" label="FILE ORGANIZATION" />

Инструменты — это половина дела. Вторая половина — конвенции, по которым команда работает в больших файлах. В командах 20+ дизайнеров без конвенций Figma превращается в мусорную свалку за квартал.

**Pages**. Минимальный рабочий набор для серьёзного проекта:

- `Cover` — титульная страница с названием проекта, превью и ссылками, которую видно в file browser.
- `Components` — библиотечные примитивы: button, input, checkbox, typography.
- `Patterns` — композиции из примитивов: форма логина, карточка товара, header с навигацией.
- `Flows` или `Screens` — production-экраны, то, что реально релизится.
- `Explorations` или `Archive` — рабочие черновики и отложенные идеи.

Эта структура разделяет стабильное от изменчивого. Components и Patterns живут долго и меняются осознанно; Screens меняются под каждую задачу; Explorations — мусорный черновик, который периодически чистится.

**Naming**. Figma использует slash-конвенцию для группировки: `Button/Primary/Large` автоматически собирается в variants grouping. Поэтому правильный паттерн — всегда slash-separated, от общего к частному: `Icon/System/16/chevron-down`, `Card/Product/Horizontal`. Это не косметика — это то, как Figma строит UI для выбора вариантов.

**Team library**. Опубликованный файл становится library, доступным во всех остальных файлах организации. Библиотека — это контракт: один раз опубликовали, все instance по проектам подтягивают изменения. Плохая привычка — публиковать слишком рано (компоненты ещё сырые, сломают все прод-файлы); хорошая — держать staging-файл, где проверяются изменения до публикации.

**Branching** — фича Figma Organization плана, появилась в 2021 году. Работает как git-ветки для файла: создаёте branch, вносите изменения, review, merge обратно. Для изменений в библиотеке это незаменимо — позволяет сделать большой refactor без риска сломать продовые файлы, пока не дойдёт до merge.

**Performance**. Figma держит граф на клиенте, и большие файлы становятся медленными. Точных порогов нет — они зависят от железа, стиля команды и количества открытых файлов, — но в практике ощутимое торможение начинается с нескольких сотен страниц и десятков тысяч нод; canvas на тысячи объектов на одной странице проседает раньше. Решение одно: делить логические домены в отдельные файлы и связывать их через team library. Marketing, product UI и internal tooling — обычно три разных файла, а не три страницы одного.

<SectionBreak id="06" label="HANDOFF И ТОКЕНЫ" />

Финальный слой — передача в разработку. Figma годами была средой, из которой разработчики пытались угадать значения по скриншотам; **Dev Mode**, анонсированный на Config 2023, это закрыл. Dev Mode — read-only интерфейс для разработчиков: они видят компоненты, значения отступов, цвета, готовые code snippets (CSS, iOS, Android) и могут пометить элемент как «ready for development».

<MarginNote>
Dev Mode — Figma Config 2023 (июнь). До этого разработчикам выдавали view-only доступ и отдельно — Inspect панель; Dev Mode объединил это и добавил code snippets.
</MarginNote>

В 2024 Figma добавили **Code Connect** — механизм, связывающий Figma-компонент с реальным кодовым компонентом в репозитории. Настроили mapping между Figma Button и React `<Button>` — в Dev Mode разработчик видит не абстрактный CSS, а готовый `<Button variant="primary" size="lg">Label</Button>`, который копируется и работает. Это переводит handoff из «перевести руками» в «вставить».

Параллельный трек — **токены**. Token Studio плагин забирает ваши Figma Variables и экспортирует их в JSON / CSS custom properties / Swift / Android resources. Формат — DTCG JSON (W3C Community Group draft, в разработке с 2019–2020). Типичный рабочий пайплайн: Figma — единственный источник, дизайнер меняет цвет, плагин экспортирует в git-репозиторий, CI собирает новую версию CSS/Swift constants, frontend подхватывает следующим деплоем. Design-to-code цикл, который раньше занимал дни переписки, сжимается до часов.

<SectionBreak id="08" label="СИНТЕЗ" />

Figma advanced — это переход от «рисовать макеты» к «строить дизайн-систему». Auto Layout делает компоненты адаптивными; Variables дают токены с типизацией, scope и modes; Component Properties и Interactive Components переводят прототип из статичного слайдшоу в живой мок с поведением; Dev Mode и Code Connect замыкают петлю с кодом.

Овладеть всем этим — не опция и не «plus on top». В команде 2026 года это минимум, ниже которого вы просто тормозите процесс: забираете время у дизайнеров, которые вынуждены вручную синхронизировать то, что Variables давно умеют, и у разработчиков, которые гадают по скриншотам вместо того, чтобы копировать готовый код. Хорошая новость в том, что это всё — механика, а не талант. Пара недель целенаправленной практики, и вы переходите из категории «знаю Figma» в категорию «владею Figma». В следующих темах раздела мы разберём, как устроен prototype-testing цикл, как строить handoff-документы, которые разработчики действительно читают, и как не сломать всё это при переходе команды с 10 дизайнеров на 30.
